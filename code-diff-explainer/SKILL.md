# Code Diff Explainer

此技能帮助开发者理解代码更改前后的逻辑变化，特别适合代码阅读能力较弱的用户。

## 核心原则

1. **先讲人话，再贴代码** — 用通俗语言描述变更，再展示具体代码
2. **先说为什么，再说是什么** — 动机提供理解代码的心理框架
3. **前后对比，而非孤立展示** — 始终展示 BEFORE 和 AFTER 的对照
4. **逐行注释** — 代码段中每一行都必须在右侧添加解释性注释，帮助代码阅读能力弱的用户理解
5. **明确说明"不做什么"** — 避免读者产生错误假设
6. **渐进式披露** — 从高层摘要逐步深入到代码细节
7. **强制逐行解释** — 对于所有代码，必须为每一行添加中文注释，确保即使完全不懂代码的用户也能理解每行的作用

## 工作流程（触发后立即执行）

**第一步：自动获取差异信息，不要等用户提醒。**

根据用户提供的信息类型，自动选择合适的方式获取 diff：

### 场景 A: 用户给了两个文件路径
- 使用 `read` 工具分别读取两个文件
- 使用 `bash` 运行 `diff` 或 `git diff --no-index file1 file2` 获取差异
- 如果文件在 git 仓库中，优先用 `git diff`

### 场景 B: 用户给了 git commit hash
- 运行 `git show <commit-hash>` 获取该 commit 的完整 diff
- 或运行 `git diff <commit-hash>^ <commit-hash>` 获取变更

### 场景 C: 用户给了两个分支或 tag
- 运行 `git diff branch1..branch2` 获取分支间差异
- 运行 `git diff --stat branch1..branch2` 先获取变更文件列表

### 场景 D: 用户给了 PR/MR 链接
- 使用 `gh pr view <number>` 或 `gh pr diff <number>` 获取差异

### 场景 E: 用户直接贴了 diff 文本
- 直接使用用户提供的 diff 内容进行解释

### 场景 F: 用户只说"解释最近的改动"
- 运行 `git diff HEAD` 查看未暂存的改动
- 运行 `git diff --cached` 查看已暂存的改动
- 运行 `git log --oneline -5` 查看最近的 commit

**第二步：分析 diff 内容**

- 识别哪些文件被修改、新增、删除
- 理解每个变更的逻辑含义
- 判断变更之间的关联性

**第三步：按下方"解释结构"输出详细解释**

> **关键提醒：** 不要等待用户提供 diff 内容，主动使用工具获取。如果用户只给了文件名或 commit hash，你自己去查。

## 解释结构

对每个代码变更，按以下结构解释：

### 1. 高层摘要（1-2 句话）

用非技术语言说明这个变更做了什么、为什么需要改。

**示例格式：**

```
这个变更添加了一个缓存层，当用户在 15 分钟内多次请求个人资料时，
从内存中读取而不是每次都查询数据库。
```

### 2. 变更动机

说明为什么要做这个改动：

- 修复了什么 bug
- 解决了什么性能问题
- 改善了什么用户体验
- 遵循了什么最佳实践

### 3. 变更详情（按逻辑分组，而非按文件）

对每个逻辑变更，使用以下模板：

```markdown
### 变更 N: [描述性名称]

**涉及文件：** `file1.ts`, `file2.ts`

**行为变化：**

- 修改前：[描述旧行为]
- 修改后：[描述新行为]

**为什么这样改：** [变更的 rationale]

**代码对比：**

> **重要：** 以下代码段中，每一行右侧都有 `//` 开头的解释性注释，说明该行代码的作用。对于代码，必须使用中文注释，确保代码阅读能力弱的用户能完全理解每行代码的含义和作用。

\`\`\`typescript
// BEFORE: [旧代码]
function oldWay() {              // 定义一个名为 oldWay 的函数
  // ...                         // 旧代码的具体逻辑
}

// AFTER: [新代码]
function newWay() {              // 定义一个名为 newWay 的函数（替换旧函数）
  // ...                         // 新代码的具体逻辑
}
\`\`\`

**关键变化点：**

1. [具体变化 1] — [为什么重要]
2. [具体变化 2] — [为什么重要]

**影响范围：** [可能影响的其他模块/功能]
**局限性：** [这个变更不处理什么]
```

### 4. 影响分析

- **风险等级：** 低/中/高 — [原因]
- **需要测试的点：** [列出需要验证的功能]
- **可能影响的下游：** [依赖此代码的其他部分]

### 5. 局限性说明

明确说明这个变更**不处理**什么：

```
注意：此变更仅处理 X 情况，不处理 Y 情况。
如果后续需要支持 Y，需要额外修改。
```

## 详细解释技巧

### 对于复杂逻辑

使用"问题-方案"模式：

```markdown
**问题：** 原来的嵌套条件语句（第 45-62 行）使逻辑难以理解。

**解决思路：** 将验证逻辑提取为独立函数 `isValidPaymentMethod()`。

**效果：**

- 主函数从 50 行减少到 15 行
- 验证逻辑可以复用
- 测试更容易编写
```

### 对于重构

强调"行为不变，结构变"：

```markdown
**重要：** 此重构不改变任何业务逻辑，仅改善代码结构。
用户看到的行为完全一致，但代码更易维护。
```

### 对于 Bug 修复

使用"症状-根因-修复"模式：

```markdown
**症状：** 用户在某些情况下看到空白页面。

**根因：** 数据加载完成前，组件尝试渲染未定义的数据。

**修复：** 添加空值检查，在数据加载完成前显示加载指示器。
```

## 代码解释示例

### 示例 1: 添加错误处理

```markdown
### 变更 1: 添加 API 错误处理

**涉及文件：** `src/api/userService.ts`

**行为变化：**

- 修改前：API 调用失败时，应用直接崩溃，用户看到白屏。
- 修改后：API 调用失败时，显示友好的错误提示，应用继续运行。

**为什么这样改：**
用户反馈在网络不稳定时应用会突然崩溃，数据丢失。
此变更确保即使 API 失败，用户也能看到有意义的错误信息。

**代码对比：**
\`\`\`typescript
// BEFORE: 直接调用，没有错误处理
async function getUser(id: string) {
const response = await fetch(`/api/users/${id}`);
return response.json(); // 如果 fetch 失败，这里会抛出未捕获的异常
}

// AFTER: 添加 try-catch 和错误提示
async function getUser(id: string) {
try {
const response = await fetch(`/api/users/${id}`);
if (!response.ok) {
throw new Error(`HTTP ${response.status}: ${response.statusText}`);
}
return response.json();
} catch (error) {
console.error('获取用户信息失败:', error);
// 向用户显示错误提示，而不是让应用崩溃
showErrorNotification('无法加载用户信息，请稍后重试');
return null; // 返回 null 让调用方知道请求失败
}
}
\`\`\`

**关键变化点：**

1. 添加 `try-catch` 包裹 — 捕获网络错误和 JSON 解析错误
2. 检查 `response.ok` — 处理 HTTP 4xx/5xx 错误（这些不会触发 catch）
3. 错误提示 — 用户知道发生了什么，而不是看到白屏
4. 返回 `null` — 调用方可以检查返回值决定下一步操作

**影响范围：**

- 所有调用 `getUser()` 的组件需要处理返回 `null` 的情况
- 需要确保 `showErrorNotification` 函数已正确导入

**局限性：**

- 此变更仅处理错误显示，不处理重试逻辑
- 如果需要自动重试，需要额外实现
```

## 注意事项

1. **避免过度自信** — 只解释代码实际做的事情，不猜测作者意图
2. **使用一致的格式** — 每次解释都使用相同结构，帮助读者建立阅读习惯
3. **分组相关变更** — 将逻辑相关的改动放在一起解释，而不是按文件顺序罗列
4. **标注严重程度** — 使用标签帮助读者聚焦重点：
   - `关键:` 必须理解的变更
   - `注意:` 可能有副作用的变更
   - `提示:` 改善代码质量但不影响行为的变更
5. **提供上下文** — 如果变更依赖其他文件或模块，说明依赖关系

## 何时使用此技能

- 用户要求解释代码变更、diff、或 commit
- 用户表示不理解某段代码为什么这样改
- 用户需要审查 Pull Request 或 Merge Request
- 用户想要学习代码重构的模式
- 用户询问"这段代码改了什么"或"前后有什么区别"
- 用户提到代码阅读困难，需要详细解释
